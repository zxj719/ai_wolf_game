# AI Is A Workflow, Not A Shortcut

AI 编程工具最容易被误解成“更快写代码”。这些笔记真正指向的结论更深：

**AI 的价值不是替人写几行代码，而是把工程推进变成可重复、可恢复、可验证的流程。**

Claude Code、OpenClaw、深度思考平台、全栈博客指南、Copilot 会话，它们表面上是不同项目，实际都在回答同一个问题：怎样让 AI 从一个聊天窗口，变成工程组织里的基础设施？

## Claude Code: From Assistant To Coworker

Claude Code 推广方案里有一个清晰定位：它不是 Copilot 式的行级补全，而是终端原生的 AI 编程代理。它能读仓库、改文件、执行命令、看结果、继续推进。

这意味着它更像一个初级到中级工程同事，而不是一个自动补全插件。既然像同事，就需要工作规则：

- 先理解仓库，再动手。
- 先提出计划，再修改文件。
- 每次改动都要能验证。
- 不确定时暴露假设。
- 遇到权限、网络、依赖问题要有清晰失败路径。
- 大型任务要拆成阶段，而不是一次性生成。

团队推广 Claude Code 的真正难点也不只是安装。安装 Node、CLI、VS Code 插件只是十分钟的事。更难的是让团队形成共同使用习惯：什么时候让 AI 做，什么时候人来判断，什么任务可以交给 agent，什么任务必须由资深工程师把关。

## OpenClaw: The Long-Running Agent Problem

OpenClaw 那份长文档讨论的是更复杂的问题：如果一个 agent 要持续推进“网络通信问题修复”这样的任务，它不能只靠一次 prompt。

长任务需要状态机。

原因很简单：工程问题会失败，会卡住，会遇到权限不足，会遇到网络异常，会发现前一个假设错误。如果没有状态机，agent 只是不断生成下一段文字；有了状态机，它才知道自己处在“观察、诊断、修改、验证、重试、人工接管”中的哪一步。

一个可用的工程 agent 至少要具备：

- 明确目标。
- 当前状态。
- 可观测信号。
- 错误分类。
- 重试次数。
- 回滚或停止条件。
- 人工接管点。
- 测试用例。

这就是 AI 工程化和普通聊天的分界线。

## Deep Thinking Platform: Productizing Attention

深度思考平台的愿景很明确：反推荐算法，打破信息茧房，聚焦价值观点。

它不是普通内容聚合器，而是一个关于“注意力质量”的产品。传统推荐系统会让用户停留更久，但不一定让用户思考更深。这个平台的方向是反过来：用 AI 整理 RSS、观点和文章，过滤噪音，提取论点，让用户看到更值得处理的信息。

技术架构已经有雏形：

- Cloudflare CDN 承载静态资源。
- Vite/React/Tailwind 构建前端。
- Workers 做 API 后端。
- D1 做数据库。
- AI API 处理摘要、观点抽取和内容整合。
- RSS 抓取作为信息入口。

真正需要继续明确的是产品边界：

- 用户每天为什么打开？
- 它和 RSS 阅读器有什么本质区别？
- AI 提取的是摘要、立场、冲突，还是行动建议？
- 如何避免它本身变成新的信息茧房？
- 如何衡量“深度思考”真的发生了？

## Full-Stack Guide: The Template Problem

全栈博客与内容聚合平台指南很长，也很有价值。它把安装、项目结构、数据库、爬虫、Cloudflare、动画、部署和 CI/CD 串起来。

但这种超长指南有一个维护问题：它越完整，越难更新。

更好的长期形态是拆成模块：

- `setup.md`：环境安装和 Claude Code 工作流。
- `architecture.md`：系统架构和数据流。
- `database.md`：D1 schema、迁移、ORM。
- `crawler.md`：RSS、Hacker News、Reddit、反封禁。
- `frontend.md`：React、Tailwind、动画和组件。
- `deploy.md`：Wrangler、GitHub Actions、生产配置。
- `troubleshooting.md`：常见失败和修复。

AI 生成长文档很容易，但工程知识最终要变成可维护模块。

## The Team Adoption Question

AI 编程工具进入团队后，会改变工程师的工作方式。它会减少一部分重复劳动，但也会暴露更多结构性问题：

- 仓库没有文档，AI 就很难理解。
- 测试不完整，AI 就不知道是否改对。
- 需求含糊，AI 会生成含糊结果。
- 权限混乱，agent 会频繁卡住。
- 没有 code review，AI 会把错误包装成自信的实现。

所以推广 AI 的本质不是“让大家都用工具”，而是逼团队补齐工程基本功。

## Working Principles

- AI agent 必须有状态机，尤其是长任务。
- 每个 AI 任务都要有明确验收标准。
- 大型指南要拆成可维护模块。
- AI 编程的核心不是 prompt，而是任务拆解。
- 工具推广要同时推广工作规范。
- AI 生成内容需要定期压缩，否则知识库会膨胀成新的噪音。
