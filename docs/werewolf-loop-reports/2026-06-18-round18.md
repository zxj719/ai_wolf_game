### 优化 1：DAY_SPEECH 女巫 identity_table

**位置**：`ROLE_DAY_SPEECH_PROMPTS['女巫']` 输出 JSON 行之前

```
【identity_table 填写指导（女巫：药水使用历史 + 备选候选记录）】
- 已救活的玩家：confidence 85-95，reason "N[X]夜银水救活（已知被狼刀目标）；若明夜再被刀：[是否还有银水/决策]"
- 已毒杀的玩家：confidence 98-100，reason "N[X]夜毒药处决确认出局"
- 高威胁候选（考虑毒）：confidence 65-85，reason "行为[带节奏/发言矛盾/攻真预]，毒药备选，威胁等级：高/中"
- 疑似关键神职（考虑下轮银水保护）：confidence 50-75，reason "行为疑似[预言家/守卫]，若明夜被刀且银水在：优先救"
```

**设计逻辑**：
- 药水使用是二元事实（confidence 85-100），比推断性条目高出一个量级
- 毒药候选（65-85）< 银水保护候选（50-75）——威胁判断比保护意图更确定
- 银水救活记录包含"若明夜再被刀"规划，让 AI 提前做二次药决策

### 优化 2：DAY_SPEECH 猎人 identity_table

**位置**：`ROLE_DAY_SPEECH_PROMPTS['猎人']` 输出 JSON 行之前

```
【identity_table 填写指导（猎人：跨轮积累开枪优先级）】
- 高威胁候选（开枪首选）：confidence 70-90，reason "开枪优先：累计[X]次可疑——[发言带节奏/投票矛盾/攻真预]，当前开枪优先级：高"
- 中等嫌疑（备选）：confidence 50-70，reason "开枪备选：[具体可疑行为]，需再观察1-2轮确认"
- 明确好人（排除出局）：confidence 15-35，reason "排除开枪对象：[原因，如'金水确认'或'发言持续可信']"
- 开枪前回顾上轮 identity_table：若某人 reason 已记"开枪优先级：高"，优先锁定
```

**设计逻辑**：
- 猎人死亡是触发开枪的条件，`HUNTER_SHOOT` case 会注入 `previousIdentityTable`（R7 验证）
- 跨轮积累意味着：N1 标注"开枪备选"，N2 补充"再次可疑"后升为"开枪优先"
- "开枪前回顾"指令确保 AI 在 HUNTER_SHOOT 时能直接利用已积累的优先级

### 优化 3：DAY_SPEECH 村民 identity_table

**位置**：`ROLE_DAY_SPEECH_PROMPTS['村民']` 输出 JSON 行之前

```
【identity_table 填写指导（村民：跨轮行为模式积累，是你唯一的持久记忆）】
- 可疑玩家：confidence 55-80，reason "行为积累：N[X][发言带节奏/投好人/回避问题]；N[Y][矛盾行为]——本轮投票对象"
- 信任玩家：confidence 15-40，reason "信任积累：多轮发言稳定，N[X]投[合理目标]，N[Y]站边正确——好人候选"
- 票型关联（疑似一伙）：confidence 60-75，reason "N[X]与[Y号]同投[Z号（被认为是好人）]——两人可能一队，同时观察"
- 每轮更新：在上轮 reason 基础上**追加**本轮新观察，不要覆盖历史——模式是多轮积累出来的
```

**设计逻辑**：
- "是你唯一的持久记忆"这句定性说明是核心——让 AI 理解为什么要认真填写
- confidence 范围：信任（低 15-40）< 票型关联（中 60-75）< 可疑（高 55-80），与评估确定性对应
- "追加不覆盖"指令最关键：没有此指令，LLM 每轮会重写 reason，抹去历史证据链

**位置**：`case PROMPT_ACTIONS.NIGHT_WITCH:` 输出行之前

```
【identity_table 填写指导（女巫夜间：药水决策状态持久化）】

- 高威胁毒药候选：confidence 65-85，reason "毒药优先候选：[发言带节奏/投好人]，本夜[是否用毒]；下轮复查"
- 确认出局玩家（已毒杀）：confidence 98-100，reason "N[X]夜毒药处决，出局确认"
```

**设计逻辑**：

- 被刀目标的 confidence（70-85）刻意低于确定药水事实（98-100），区分"已知狼刀"与"已知毒杀"两种确定性

## 4. 发现的潜在 bug（写入本轮修复过程）

**T20 捕获的模板插值漏洞**：在编写 NIGHT_WITCH 夜间指导时，最初写了：
```
reason 写"...解药已${canSave ? '可用' : '已用'}；..."
```
由于 NIGHT_WITCH case 返回语句是模板字符串，`${canSave ? ...}` 会在运行时被求值，导致 AI 在 `canSave = false` 时读到"解药已已用"（语法重复）。测试 T20 专门捕获了这个问题，已修复为静态描述文本。

这提醒我们：在 NIGHT_* case 内部的模板字符串中，写指导文本时须避免使用已声明的变量名（如 `canSave`、`dyingId`）作为 `${}` 表达式，否则会变成运行时动态值而非静态指导。
