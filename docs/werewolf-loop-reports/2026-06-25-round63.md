## 问题识别与修复

### 骑士 DAY_VOTE 读写闭环缺口（R62 建议，本轮执行）

**问题**：骑士在 knight.js `getKnightDaySpeechPrompt` 中（R44 补全）写入"决斗候选：[优先级A/B/C]，[逻辑依据摘要]"到 identity_table，但 DAY_VOTE 走通用分支：`"有查杀 → 跟投查杀！无查杀 → 投发言逻辑最混乱..."` — 通用分支不读 identity_table，骑士多轮积累的决斗候选在投票阶段被完全忽略。

**DAY_VOTE 闭环矩阵对比**：
```
修复前：
  knight.js DAY_SPEECH写"决斗候选" → DAY_VOTE（通用，不读）
                                              ↑
                                            断链点

修复后：
  knight.js DAY_SPEECH写"决斗候选" → DAY_VOTE（骑士专属框架，读✅）
                                    完整闭环 ✅
```

**博弈论依据**：
- 骑士的投票和决斗是**互补的顺序工具**：若决斗候选可被群体投票出局，投票更省资源（不消耗一次性决斗能力）
- 若无法翻转票型，仍然投向决斗候选能积累好人阵营的公共认知，为后续决斗建立公信力
- 已使用决斗后骑士身份公开，应充分利用 1.5 倍票权作为判断锚点带动票型
- 这是"工具互补性"原则在竞技狼人杀中的应用（Wang 2025 §3.2 策略工具稀缺性管理）

**修复**（`src/services/aiPrompts.js`，DAY_VOTE case）：

1. 新增 `knightHasDueledForVote` 和 `knightVoteStrategy` 常量（双状态）：
```javascript
// 骑士投票策略：DAY_SPEECH积累的"决斗候选"与投票方向对齐（vote-duel对齐原则，R63）
const knightHasDueledForVote = currentPlayer?.hasUsedDuel ?? false;
const knightVoteStrategy = knightHasDueledForVote
    ? `2. 【骑士投票—领袖框架（身份已公开）】决斗已使用，你是场上公信力锚点：...`
    : `2. 【骑士投票—决斗候选对齐框架（技能待用）】
   a) 先查 identity_table 中 reason 含"决斗候选"的存活玩家 ... (vote-duel 对齐)
   b) 场景评估：能推票出局 → 全力推票（保留决斗）；无法翻转 → 仍然投他积累认知
   c) 无"决斗候选"标注时：跟投查杀 > 逻辑崩塌 > 历史热力`;
```

2. 在 playerRole 链条插入骑士分支（猎人→骑士→通用）：
```javascript
  : playerRole === '猎人'
  ? hunterVoteStrategy
  : playerRole === '骑士'
  ? knightVoteStrategy
  : `2. 【投票策略】有查杀 → ...通用...`
```
