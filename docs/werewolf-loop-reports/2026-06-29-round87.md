## 问题背景

### R77 里程碑声明与实际状态的差距

R77 LEARNINGS.md 记录："DAY_SPEECH 全角色个性化完成里程碑：R77 后，所有非平民神职角色（预言家/守卫/女巫/猎人/骑士/摄梦人/魔术师）和狼人的 DAY_SPEECH 均有 personalityLens。"

但实际检查 `knight.js`：

```javascript

export const getKnightDaySpeechPrompt = (ctx, params) => {
  const { hasUsedDuel, aliveCount: aliveNow = 8 } = params;
  // ... 没有 personalityType 读取，没有 personalityLens 变量
```

R77 里程碑声称覆盖了"骑士"，但实际上 R77 只在 `aiPrompts.js` 中完成了摄梦人/魔术师的 personalityLens（这两者在各自模块文件中实现）。骑士的 DAY_SPEECH 委托给 `getRoleModule('骑士').daySpeech(ctx, params)`，需要在 `knight.js` 中实现，而非在 `aiPrompts.js` 中内联。

### 骑士 personalityLens 的设计挑战

骑士的 personalityLens 设计比其他神职更复杂，因为骑士有两种截然不同的行为模式：

| 阶段 | 状态 | 行为目标 | lens 影响维度 |
|------|------|---------|-------------|
| pre-duel（隐藏期）| `hasUsedDuel = false` | 伪装成普通村民，积累决斗证据 | 如何在掩护下建立决斗证据链 |
| post-duel（领袖期）| `hasUsedDuel = true` | 身份已公开，作为权威领袖指挥好人 | 如何用验证后的公信力带动全场 |

这使得每个 personality 分支需要两条不同指令——一条给 `hasUsedDuel = false`（隐藏期策略），一条给 `hasUsedDuel = true`（领袖期策略）。

## 实施内容

### 新增变量（在 `endgameNote` 之后，`knightHistoryStep` 之前声明）

```javascript
// R87：骑士 DAY_SPEECH 个性化发言风格（pre-duel 隐藏期 + post-duel 领袖期双阶段）
const knightPersonalityType = params.personalityType || '';
let knightPersonalityLens = '';
if (knightPersonalityType === 'logical' || knightPersonalityType === 'analytical') {
  knightPersonalityLens = hasUsedDuel
    ? '\n【你的领袖风格】数据驱动型：...'
    : '\n【你的发言风格】推理积累型：...';
} else if (knightPersonalityType === 'aggressive') {
  // ... 7 分支 × 2 阶段
}
let knightSpeechLen = '40-80字';
if (knightPersonalityType === 'aggressive') knightSpeechLen = hasUsedDuel ? '30-55字' : '35-55字';
else if (knightPersonalityType === 'cautious') knightSpeechLen = '55-90字';
```

### 7 种个性类型双阶段策略总结

| 类型 | pre-duel 策略 | post-duel 策略 |
|------|-------------|--------------|
| logical/analytical | 推理积累型：量化置信度框架，逻辑链伪装 | 数据驱动型：决斗验证→推导逻辑体系带全场 |
| aggressive | 速攻伺机型：达到下限即刻翻牌，简短有力 | 快速主导型：直接宣告方向，行动可信度压制异议 |
| emotional | 直觉信号型：直觉补充逻辑，感性掩护骑士视角 | 感染共情型：分享决斗心理过程，感染好人紧跟 |
| contrarian | 反预判型：质疑主流判断，保留独立分析价值 | 独立判断型：补充漏网次级威胁，防节奏被操控 |
| cunning | 伏击谋略型：埋设引导让目标自我暴露，最高价值时出手 | 战略释放型：战略分配调查方向，保留判断制造压力 |
| cautious | 充分准备型：严格按阈值，多倾听少指名 | 严谨指挥型：详细说明决斗逻辑，确保全场共识后带票 |
| steady | 稳健蓄势型：先肯定共识再提异议，稳中求准 | 协调渐进型：协调者而非独裁者，平衡各角色配合 |

## 关键设计决策

### 双阶段统一 `if-else` 分支（不分离函数）

选择在每个分支内用 `hasUsedDuel ? '...' : '...'` 三元表达式处理两个阶段，而非创建两组独立的变量（pre-duel lens 和 post-duel lens）。

**原因**：
1. 同一个 personality 的两个阶段共享同样的"行为倾向"，三元表达式明确表达了这种连续性
2. 避免引入两倍的变量（14 个分支 vs 7 个），减少代码复杂度
3. 与 `aggressive` 的 `knightSpeechLen` 计算（同样用 `hasUsedDuel` 三元）保持一致

### `aggressive` post-duel speechLen 更短（30-55字）

post-duel 的 aggressive 骑士比 pre-duel（35-55字）还要更短。理由：身份公开后，aggressive 骑士直接宣告"投X号"，不需要任何铺垫。这是符合 `aggressive` 类型"结论先行省略铺垫"原则的自然延伸。
