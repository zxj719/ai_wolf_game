## 本轮优化内容

### 1. promptFactory.js 整体删除（-282 行）

**背景**: R2（2026-06-15）发现 `promptFactory.js` 中 `getProgressiveActionPrompt` 从未被任何生产代码 import，导致其调用的 `rolePrompts/*.js` 中的 nightAction/daySpeech 函数也全部失效（形成死代码级联）。R14 删除了六个角色的 dead 函数，R52 再次清理 14 个 dead 函数。但 `promptFactory.js` 本身作为死代码入口一直未删除。

**确认方法**:
```bash
grep -rn "promptFactory\|getProgressiveActionPrompt\|buildProgressivePersonaPrompt" src/ --include="*.js" --include="*.jsx" | grep -v "src/services/promptFactory.js"
# → (no output) 确认无任何外部引用
```

**删除内容**: 整个 `src/services/promptFactory.js` 文件（282 行）

**意义**: 彻底封闭 R2 教训记录的死代码入口。未来新开发者无法再错误地认为"修改 rolePrompts/*.js 的 nightAction/daySpeech 函数就能改变 AI 行为"。

### 2. 预言家对跳 few-shot 示例（R26+R19 建议合并执行）

**背景**: R26（2026-06-19）添加了对跳三步法 Step A→B→C 框架，但缺少具体的 before→after 格式示例。R19 教训明确：涉及格式转换的指导，必须附上具体示例。

**修改位置**: `aiPrompts.js` 的 `ROLE_DAY_SPEECH_PROMPTS['预言家']` case 中，`counterClaimSection` 变量，Step C 行后，`【语气原则】` 前。

**新增内容**:
```
【执行示例（few-shot）】假设你是真预言家，N1验3号=好人，N2验7号=狼人，对方${ccIds}号报3号=狼人：
✗ 错误写法："我是真预言家，对方是悍跳狼，请相信我。"（只有声明，没有信息）
✓ 正确写法（Step A→B→C 框架）："我的查验记录：N1夜3号好人、N2夜7号狼人（Step A）。${ccIds}号报3号是狼人——但我已验3号是好人，两份报告直接矛盾（Step B）。我N2选验7号是因为N1他站边了最终出局的狼人——有完整的查验逻辑链。请${ccIds}号解释：为什么选择验3号？验之前有什么判断依据？（Step C）"
```

**设计原则**:
- before→after 对比：让 LLM 精确理解"没有框架"和"有框架"的输出差异
- 示例体现了 Step A（记录）→ Step B（矛盾点）→ Step C（心路历程提问）完整链
- `ccIds` 动态变量在示例中合理出现（是对跳玩家号码的占位），符合 R18 规则（指导文本静态部分无变量，示例中的动态值用有意义的命名变量）

**同步路径**:
- `aiPrompts.js` ROLE_PERSONAS 降级路径：`thinkingDimensions[2]` 更新为含 Step A/B/C
- `seer.js` `getSeerThinkingDimensions` dimension[1]：更新为三步法描述（含 Step A/B/C 明确标注）
