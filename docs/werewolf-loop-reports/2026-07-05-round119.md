## 改动细节

### 旧 seerVoteStrategy（2路径，约240 chars）

```javascript
// 预言家投票策略：对跳局面时悍跳者优先于其他目标
const seerVoteStrategy = seerCounterClaimantsInVote.length > 0
    ? `2. 【预言家投票策略—对跳优先】...绝不投金水（已验好人）`
    : `2. 【预言家投票策略】率先投查杀目标！绝不投金水（已验好人）。引领好人集中票型形成多数。`;
```

问题：
- 无具体查杀/金水 ID → AI 只能 NLP 猜测，而非从结构数据读取
- "绝不投金水" 是负向禁令（白熊效应风险）
- 无 confidence=100 锚点建立 identity_table 高权重

### 新 seerVoteStrategy（3路径 + 5个辅助变量，R119）

```javascript
// 新增5个辅助变量
const dvSeerMyChecks = playerRole === '预言家'
    ? (gameState.seerChecks || []).filter(c => c.seerId === currentPlayer?.id)
    : [];
const dvSeerConfirmedWolves = dvSeerMyChecks
    .filter(c => c.isWolf && alivePlayers.find(p => p.id === c.targetId))
    .map(c => c.targetId);
const dvSeerConfirmedGood = dvSeerMyChecks
    .filter(c => !c.isWolf && alivePlayers.find(p => p.id === c.targetId))
    .map(c => c.targetId);
const dvSeerKillSummary = dvSeerConfirmedWolves.length > 0
    ? dvSeerConfirmedWolves.map(id => `${id}号`).join('、')
    : '暂无存活的查杀目标';
const dvSeerGoodSummary = dvSeerConfirmedGood.length > 0
    ? dvSeerConfirmedGood.map(id => `${id}号`).join('、') + '（已确认好人，从投票选项中排除）'
    : '暂无';

// 三路径策略
const seerVoteStrategy = seerCounterClaimantsInVote.length > 0
    ? `2. 【预言家投票策略—对跳优先（R119 读写闭环）】
   a) 🔮查验锚点：已验查杀→存活：${dvSeerKillSummary}；已验金水→存活：${dvSeerGoodSummary}
   b) ${seerCounterClaimantsInVote.join(',')}号是在场悍跳者...
   c) 投票排序：① 你查杀的存活狼人（一手验证最高） → ② 悍跳者 → ③ identity_table 逻辑崩塌者`
    : dvSeerConfirmedWolves.length > 0
    ? `2. 【预言家投票策略—查验锚点导引（R119 读写闭环）】
   a) 🔮查验锚点：已验查杀→存活：${dvSeerKillSummary}；已验金水→存活：${dvSeerGoodSummary}
   b) 投票优先排序：① 已验查杀的存活狼人（confidence=100）→ ② identity_table 最高嫌疑 → ③ 崩塌者
   c) 领袖义务：率先带票投查杀方向，引导好人集中形成多数`
    : `2. 【预言家投票策略】率先投查杀目标！已验为好人的玩家从投票选项中排除（...）。引领好人集中票型形成多数。`;
```

**设计依据**：
- 直接读 `gameState.seerChecks` 结构数据 >> 让 AI 从 identity_table 字符串中 NLP 解析（前者是 ground truth，后者是 AI 推理结果可能出错）
- `confidence=100` 明确告知 identity_table 中应有最高权重，避免 AI 自行降权
- 领袖义务描述符合预言家作为信息最充分角色的博弈定位（Wang 2025 arxiv:2408.17177 的信息优势理论）

## 本轮教训（→ LEARNINGS.md）

1. **seerVoteStrategy 三路径设计是"对跳 > 有查杀 > 无记录"**：与猎人/骑士"有锚点 > 无锚点"的双路径相比，预言家多了"对跳优先"这第一层过滤，必须在对跳路径中同时展示查验锚点（信息复用）。
2. **金水禁令从负向改为正向**："绝不投金水" → "已确认好人，从投票选项中排除（confidence 最高，维持队友信任）"。后者是第 34 次应用白熊效应正向铁律（R112 第 33 次）。
3. **DAY_VOTE 窗口必须覆盖输出JSON格式行**：`输出JSON格式:` 在 DAY_VOTE block 中位于 offset ~13000+ 位置，任何向 block 新增 >1000 chars 的操作都会推动该行超出 12000-char 窗口。铁律：新增操作后必须检查 R71 窗口。
