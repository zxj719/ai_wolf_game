## 2. 问题诊断

### 根因："感知-执行分裂"在 pkHint 层面的延伸

Round 34 修复了**狼人**的 pkHint 问题（进攻框架在防御场景失效）。但 Round 34 的通用 pkHint 对**预言家**和**守卫**同样存在"感知-执行分裂"：

**通用 pkHint 的不足：**
```
- 提供新论点（不要重复之前已说过的话……）
- 直接回应或质疑对方……
- 明确陈述你存活的价值（不要泛说"我是好人"，说明你有什么未完成的信息/判断/保护……）
```

**对预言家的问题：**
- "陈述存活价值"对预言家是**可量化的**（还有几夜查验能力、信息链的延续），但通用框架没有告诉 AI 怎么量化
- PK 是公开**所有**查验记录的最佳时机（包括之前可能保留的），通用框架没有指出这个时机优势
- 如果对手是悍跳者，**Step A-B-C** 框架是最强的 PK 武器，但通用框架不知道要整合这个分支

**对守卫的问题：**
- 守卫在 PK 中面临**独特的二元决策**：暴露身份赢 PK 但成为夜间目标；保持隐秘保留守卫价值但 PK 竞争力弱
- 通用框架的"陈述存活价值"对守卫是**两条路**，而不是一个点
- 残局（≤5人）和非残局对守卫的最优策略完全不同（残局：暴露；非残局：评估保护资源后决定）

## 3. 技术实现

### 3.1 `aiPrompts.js` — pkHint 从三元表达式改为 `let + if` 多分支块

**改前（Round 34）：**
```javascript
const pkHint = roleParams.pkMode
    ? `\n\n【⚔️ PK 辩护模式（平票决赛）】你与其他候选人进入最终对决！……`
    : '';
```

**改后（Round 35）：**
```javascript
let pkHint = '';
if (roleParams.pkMode) {
    if (playerRole === '预言家') {
        const hasCC = (roleParams.counterClaimants || []).length > 0;
        const ccLine = hasCC
            ? `- 对手有悍跳背景：执行 Step A（报完整记录）→ Step B（找对方报告矛盾）→ Step C（解释查验心路历程）——真假预的核心区别是决策逻辑，而非结论`
            : `- 提供新论点：PK 发言必须包含之前未说过的新信息或新判断——重复旧话等于说服力归零`;
        pkHint = `\n\n【⚔️ PK 辩护模式（平票决赛）—— 预言家专属框架】
- 部署全部查验：PK 是公开所有查验记录的最佳时机（含之前保留的）……
- 量化存活价值：明确指出你的存活对好人阵营的意义……
${ccLine}`;
    } else if (playerRole === '守卫') {
        const aliveNow = roleParams.aliveCount || 8;
        const phaseLabel = aliveNow <= 5
            ? '（残局）：暴露身份收益较高……'
            : '（非残局）：若场上有预言家/女巫等保护力量……';
        pkHint = `\n\n【⚔️ PK 辩护模式（平票决赛）—— 守卫专属框架】
核心权衡：暴露守卫身份 vs 继续隐秘，取决于局面价值：
- 当前存活${aliveNow}人${phaseLabel}
- 若暴露身份：报出守护记录（可核实），说明守卫对好人阵营的持续保护价值
- 若保持隐秘：聚焦逻辑分析，找出对手发言中的具体矛盾，用清晰推理建立信任
- 无论哪条路：必须有之前未说过的新论点或新判断……`;
    } else {
        // 通用框架（狼人/女巫/猎人/村民/骑士/摄梦人/魔术师）
        pkHint = `\n\n【⚔️ PK 辩护模式（平票决赛）】……（原有三要点）……`;
    }
}
```

**设计要点：**
- `let + if` 块（遵循 R15：多分支不用三元）
- `ccLine` 和 `phaseLabel` 预先计算（遵循 R18：不在模板字符串内写内联三元/boolean 插值）
- `playerRole` 在 `generateUserPrompt` line 1214 已声明，在 DAY_SPEECH case 内可访问
- `roleParams.counterClaimants`（已有，预言家专属）和 `roleParams.aliveCount`（已有，全角色）可直接使用
- 通用 `else` 分支完整保留原有三要点（零回归风险）

### 3.2 数据依赖（无需改动调用端）

| 数据 | 来源 | 说明 |
|------|------|------|
| `roleParams.counterClaimants` | `generateUserPrompt` line 1256-1260 | 预言家专用，已有 |
| `roleParams.aliveCount` | `generateUserPrompt` line 1254 | 全角色，已有 |
| `playerRole` | `generateUserPrompt` line 1214 | 全局可访问，已有 |

`useDayFlow.js` 的 `handlePKRound` 传 `{ pkMode: true, pkCandidates: pkCandidateIds }` 不需要变动，所有需要的数据从 `gameState` 和 `players` 中在 case 块内自动计算。
