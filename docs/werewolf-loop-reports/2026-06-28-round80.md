### 新增条件化平安夜推断步骤

**文件**：`src/services/aiPrompts.js`（`ROLE_DAY_SPEECH_PROMPTS['村民']` 函数内）

**新增变量**（`villagerSpeechLen` 赋值之后、`return` 之前）：

```javascript
// R80：平安夜推断框架（D2+适用：利用夜间无死亡信息更新 identity_table）
const isPeacefulNight = ctx.dayCount > 1 && ctx.lastNightInfo?.includes('平安夜');
let peaceNightStep = '';
if (isPeacefulNight) {
    const prevDay = ctx.dayCount - 1;
    peaceNightStep = `⭕【平安夜推断（仅在 thought 中分析；speech 只说"平安夜，继续分析局势"即可）】
- 查【投票记录】D${prevDay}中票压最高的存活玩家——是昨夜最可能被刀但未死的人
- 若场上有守卫或女巫：该玩家 identity_table confidence 可降 10-20（被狼重点针对=更可能是好人）
- 若昨日票型分散（无明显高票目标）：平安夜信息量有限，维持现有判断不轻易调整\n`;
}
```

**模板注入**（Step0 后、Step1 前）：

```
Step0: 【读取历史推理积累...
${peaceNightStep}Step1: 基于已有信息，我最怀疑谁？证据是什么？
```

- 非平安夜：`peaceNightStep = ''`，模板直接输出 `Step1:`，无任何空行
- 平安夜（D2+）：在 Step0 后、Step1 前注入完整推断步骤

## 设计依据

### 感知-执行分裂问题

`ctx.lastNightInfo` 含"平安夜(无人死亡)"，通过 `getBaseContext(ctx)` 的"昨夜情况"字段已注入 base prompt。  
`ctx.voteInfo` 含 `D1:1->2,...=>result` 格式的全局投票历史，同样通过 base prompt 注入。

两个信息都"可见"，但村民 AI 没有脚手架知道：
1. 票压最高的存活玩家 = 最可能的狼刀目标（被守护或女巫救了）
2. 这个信息应该降低其 confidence（更可能是好人，因为狼重点针对）

本轮修复将推断链条显式化。

### 三分支推断逻辑

| 场景 | 推断逻辑 |
|------|---------|
| 有守卫或女巫 + 有高票目标 | 高票存活玩家 confidence 降 10-20（被护成功的好人候选） |
| 票型分散（无高票目标） | 平安夜信息量有限，维持现有判断 |
| D1 或非平安夜 | peaceNightStep = ''，不注入（完全回退） |

### Speech 约束

步骤限定"仅在 thought 中分析；speech 只说'平安夜，继续分析局势'即可"，避免村民 AI 在发言中暴露过于复杂的推断过程，破坏"普通村民"的发言风格。

## 新增教训

### R80-A：感知-执行分裂检验原则
每个"AI 能看到的信息"都应对应"AI 知道怎么用这个信息"的推断步骤。`getBaseContext` 注入的 `lastNightInfo` 和 `voteInfo` 是纯感知输入，无脚手架=感知-执行断层。修复方式：在相关步骤之间插入条件化推断步骤。

### R80-B：ROLE_DAY_SPEECH_PROMPTS block 定位
测试定位某角色 block 时，不要猜测下一个角色是哪个——用 `src.indexOf()` 验证。村民后是骑士（`'骑士':`），不是猎人（猎人没有 DAY_SPEECH 函数）。

### R80-C：T18 白盒测试设计
验证"模板使用变量插值而非硬编码"：不能用 `expect(block).not.toContain(stepContent)` 因为 peaceNightStep 赋值语句本身在 block 内。应切片 `return \`` 后的模板段，在其中验证 `${peaceNightStep}` 存在且 `⭕` 不存在。
