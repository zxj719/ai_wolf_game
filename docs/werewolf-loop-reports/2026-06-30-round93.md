## 本轮优化内容：骑士领袖期历史读取 Step0 ④（R93）

### 问题识别（感知-执行分裂）

R89 添加了 `knightLeaderStep`（Step0.5），在领袖期每天执行：
- ① 读取决斗锚点 → 连锁推断
- ② 三选一战略框架（集火型/调查型/保护型）
- **③ 写入**：`D${ctx.dayCount}领袖指令：[战略A/B/C]，[本轮核心目标玩家号]` 到 identity_table

identity_table 写指导也标注："（供下轮阅读领袖行动历史）"

但 `knightHistoryStep`（Step0）只读取 `"决斗候选"` 字样的条目：
- 路径A：候选存活 → 更新优先级
- 路径B：候选已投票出局 → 触发续战搜索  
- 路径C：候选已被夜杀 → 触发续战搜索

**无任何读取 `D{n}领袖指令` 的子步骤** → AI 在次日能看到 identity_table 中的领袖指令记录，但无推断脚手架指导其如何读取和使用 → 感知-执行分裂。

### 修改内容

**文件**：`src/services/rolePrompts/knight.js`

**新增变量** `knightLeaderReadHint`（在 `knightHistoryStep` 之前声明）：

```javascript
// R93: 骑士领袖历史读取（DAY→DAY 闭环补完：Step0 ④ 读取上轮领袖指令，修复感知-执行分裂）
const knightLeaderReadHint = (hasUsedDuel && ctx.dayCount > 1)
  ? `   ④ 读取上轮领袖指令（thought 中完成）：查 identity_table 中含"D${ctx.dayCount - 1}领袖指令"字样的条目：
      路径A（上轮集火型）：核心目标当前存活 → 今日继续集票带节奏；目标已出局 → 续战切换新集火目标
      路径B（上轮调查型）：回顾上轮交叉验证结果，将新确认的玩家立场整合进今日战略选择
      路径C（上轮保护型）：查关键好人/神职今日是否仍有威胁；仍危险 → 延续战略C；已安全 → 今日更新战略
      读取完成后在 thought 中明确今日沿用上轮战略还是切换（并简述原因）\n`
  : '';
```

**修改 `knightHistoryStep`** 末尾：在 ③ 后追加 `\n${knightLeaderReadHint}`，使 ④ 成为 Step0 的第四子步骤。

### 激活条件设计

- `hasUsedDuel && ctx.dayCount > 1`：只有领袖期（hasUsedDuel=true）且 D2+ 才需要读取历史
- D1 骑士（未决斗）：`knightHistoryStep` 走 D1 fallback 分支，hint 本身 `dayCount > 1` 不满足，双重保护
- 非领袖期（hasUsedDuel=false）：hint 不激活，向下兼容

## 博弈论依据（Wang 2025，arxiv:2408.17177）

骑士领袖期的战略连续性对 AI 游戏质量影响巨大：
- **集火型**：一旦领袖决定集火 X 号，次日应维持该判断而非重新评估，否则集火节奏会分散
- **调查型**：上轮收集的交叉验证结果是本轮的起点，不读取则等于"遗忘"
- **保护型**：若关键好人已安全，领袖应及时更新战略而非僵化延续
- 这三种路径的共同点：**领袖效能 = 历史战略积累的利用率**，无读取等于零利用率
