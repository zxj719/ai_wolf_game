## 问题背景

### 旧版 `knightHistoryStep` 的感知-执行分裂

```javascript
// 旧版（R44 实现，R51 后未变）
const knightHistoryStep = ctx.dayCount > 1
  ? `0. 【读取历史决斗候选】查看系统提示中【你之前的身份推理表】：哪些玩家的 reason 含"决斗候选"字样？
   - 确认这些候选目标是否仍然存活（已出局则跳过）   ← 只有"跳过"，无替补框架
   - 结合今天新信息（发言漏洞/死亡验证/验人结果）更新候选优先级
   - 历史候选是决策起点，不是硬约束——若新信息推翻旧候选，更新并在 thought 中说明原因`
  : '...';
```

**问题**："已出局则跳过" 是一个空操作——骑士知道候选不在了，但没有任何框架指导如何寻找下一个候选。这是典型的感知-执行分裂：信息传达了（知道候选出局），但执行路径缺失（不知道怎么办）。

这个问题在以下场景中尤为突出：
- **场景1**：骑士在 D2 积累了"优先级A 决斗候选"，但 D2 群体将该目标投票出局（好人方向一致），D3 骑士的决斗资源被"节省"了，但旧框架没有指导骑士如何利用这个信息更新局势评估
- **场景2**：骑士的决斗候选被狼人夜杀，说明该候选是好人（反向验证），但旧框架只说"跳过"，错失了更新 identity_table confidence 的机会

## 实施内容

### 新版 `knightHistoryStep`（三步三路径框架）

```javascript
// R86: 三路径评估 + 续战搜索框架
const knightHistoryStep = ctx.dayCount > 1
  ? `0. 【读取历史决斗候选与续战策略（thought 中完成）】
   ① 查 identity_table 中含"决斗候选"字样的玩家，确认其当前状态，分三路径处理：
      路径A（候选存活）：结合今天新信息（发言漏洞/死亡验证/验人结果）更新候选优先级，正常进入决斗评估流程
      路径B（候选已被投票出局）：好人阵营已消灭该目标，决斗资源节省；identity_table 追加"→已投票出局（好人方向一致）" → 触发续战搜索
      路径C（候选已被狼夜杀）：目标已确认为好人，原决斗线索丢失；identity_table 追加"→已被狼击杀（铁好人确认）" → 触发续战搜索
   ② 续战搜索（路径B/C 触发）：按优先级 A > B > C 在存活玩家中重新扫描：
      决斗优先级A（阈值≥${thresholdA}%）：场上是否有新的预言家对跳局面？
      决斗优先级B（阈值≥${thresholdB}%）：是否有可疑假金水玩家需要破除？
      决斗优先级C（紧急救场）：是否有即将被错误投出的真神职需要搏命救场？
      将首个满足阈值的候选 reason 追加"→重启决斗候选：[优先级A/B/C]，[新依据]"
   ③ 历史候选是决策起点，不是硬约束——若新信息推翻旧候选，更新并在 thought 中说明原因`
  : '0. 【第一天】无历史决斗候选记录——直接根据当前局势推断决斗候选目标';
```

### 新 identity_table 关键词（新增 1 行）

```
- 候选已出局更新：追加"→已投票出局（好人方向一致）"（路径B）或"→已被狼击杀（铁好人确认）"（路径C）；新候选找到后追加"→重启决斗候选：[优先级A/B/C]，[新依据]"
```

## 三路径设计依据

| 路径 | 触发条件 | 推断价值 | 执行动作 |
|------|---------|---------|---------|
| A（候选存活）| 候选仍在局 | 更新优先级，无变化 | 正常决斗评估流程 |
| B（候选被投票出局）| 候选被群体淘汰 | 好人方向一致（正反馈），决斗资源节省 | identity_table 追记 + 触发续战搜索 |
| C（候选被狼夜杀）| 候选被狼夜杀 | 目标确认为好人（反向验证），提高该玩家 confidence=90+ | identity_table 追记 + 触发续战搜索 |

**续战搜索与 R51 终局阈值自然整合**：
- 续战搜索中使用 `${thresholdA}` / `${thresholdB}`（残局时下调为 50/40，正常时为 70/60）
- `thresholdA` / `thresholdB` 在 `knightHistoryStep` 声明之前已定义（R51 添加），声明顺序安全

**竞技依据（Wang 2025，arxiv:2408.17177）**：
LLM 骑士的最大弱点之一是"单点依赖"——整局只有一个决斗候选，该候选一旦消失就陷入决策真空。三路径框架使骑士的决策链具备韧性：任何路径的候选变化都能触发对应的更新动作。
