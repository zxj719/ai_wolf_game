## 2. 本轮改进

### 感知-执行分裂修复：DAY_VOTE 本轮发言票型感知

**问题根因**：
- DAY_VOTE prompt 不使用 `getBaseContext(ctx)`，因此 `todaySpeeches` **从未出现**在 DAY_VOTE 上下文中
- `speechHistory.voteIntention` 是结构化字段，但从未被汇总到投票阶段
- 狼人有"防守局面"执行路径，但无任何感知信号驱动激活 → 经典**感知-执行分裂**

**修复内容**（`src/services/aiPrompts.js`，DAY_VOTE case）：

#### A. 本轮发言票型摘要（所有玩家可见）

从 `speechHistory.voteIntention` 提取当天已表达的投票意向：
```
【本轮发言票型】已有N人表达投票意向：X号(n票意向)、Y号(m票意向)。
```
- 排除当前玩家自身的意向（避免自引用）
- 排除弃票（voteIntention = -1）
- 按票数降序排列

**价值**：所有角色（猎人、预言家、骑士、村民）都能看到当前轮的投票趋势，更好地决定是否跟投或对抗。

#### B. 狼人防守局面感知信号（仅狼人）

在狼人投票框架的 a) 刀口对齐之后，注入预计算的局势诊断：

**触发条件**：已表态玩家中，有 ≥ ceil(N/2) 人指向同一个狼人队友
```
⚡【防守局面已触发】已表态的N人中，n人目标指向你队友X号 → 执行"防守局面"！
```

**未触发但有压力**（少量票意向指向队友）：
```
【局势预警】队友承受票压：X号(n票意向)，评估是否接近触发"防守局面"阈值。
```

**注入位置**：紧跟在 `a) 刀口对齐` 之后、`b) 场景选择` 之前，确保 wolf 读完刀口信息即看到防守信号，自然引导其选择正确场景。

## 4. 关键决策备忘

### 为什么不在 DAY_SPEECH 阶段就注入票型感知？
DAY_SPEECH 阶段是序列化的（按顺序发言），后说话的 wolf 可以从对话上下文中看到前面玩家的发言（包括声明的投票意向）。DAY_VOTE 则是并发决策——所有发言完成后才进入投票，AI 只接收单轮 prompt，没有之前发言的自然上下文。因此 DAY_VOTE 才是真正的感知盲区。

### 为什么阈值选 ceil(N/2) 而非 N/3？
- N/3 太敏感：2人中1人指向队友就触发，容易误触发
- ceil(N/2) = 严格多数：需要超过一半的已表态者才触发，与游戏中"票型领先=追杀成功"的临界点对齐
- 预警（少量票压）用于 N/3 < x < ceil(N/2) 区间，提供梯度感知

### DAY_VOTE 为什么没有 getBaseContext(ctx)？
DAY_VOTE 的信息需求与 DAY_SPEECH 不同——投票时不需要"发言模板/不重复要求"，需要的是"结构化决策框架+局势数据"。添加 getBaseContext 会把重复性发言规则带入投票，增加干扰。因此本轮用提取 voteIntention 字段的方式精准补充票型信息，而非整合全部发言文本。
