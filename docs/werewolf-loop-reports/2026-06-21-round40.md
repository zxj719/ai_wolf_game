## 2. 问题诊断

### 根因：NIGHT_GUARD identity_table 的读写闭环缺失

**发现方式**：R39 LEARNINGS 规则 item 85 人工预标注，R39 诊断命令未能捕获（NIGHT_GUARD 写指导无"下轮复查"关键词，只有"守护优先级：高/中"标记）。

**信息流断裂（同 R38/R39 模式）**：
```
N1 夜写：reason = "行为疑似预言家，守护优先级：高"  ← 写入 identity_table（R17 加入）
N1 夜后：identity_table 存入 identityTablesRef      ← 正常存储
N2 夜始：系统提示注入 previousIdentityTable         ← 正常传递
N2 夜思维链：【守护优先级】已跳身份的预言家 > 重要神职  ← 静态列表，从不读取 identity_table 中的历史标记
                                                   ← 感知-执行分裂
```

**守卫特殊性**：与 WOLF/WITCH 不同，守卫有"禁止连守"约束（`cannotGuard` 参数），历史优先候选恰好是昨夜守护目标时需要顺延。Step 0 因此额外提示"若该目标在禁止连守限制内，改选次高优先候选"；Step 2 专门处理此顺延逻辑。

**修复后**：
```
N2 夜 Step 0："查看系统提示中【你之前的身份推理表】：哪些玩家的 reason 含'守护优先级：高'
              或'守护优先级：中'？将其列为今晚守护候选起点
              （若该目标在禁止连守限制内或已出局，改选次高优先候选）"
                                                   ← 显式读取指令 + 连守禁令处理，形成闭环
```

**诊断命令状态**：本轮修复后，`grep "供下轮\|下轮用\|下轮参考\|下轮复查" aiPrompts.js` 首次返回空——代表 WOLF/WITCH/GUARD 三个夜间行动的读写闭环均已完成。

**变更 2：新增两个 const（R18 规范 + 读写闭环）**
```javascript
// R18 规范：避免在 return 模板字符串内直接使用 ctx.dayCount
const guardNightLabel = `N${ctx.dayCount}`;
// 读写闭环（同 R38/R39 NIGHT_WOLF/WITCH 模式）
const guardHistoryStep = ctx.dayCount > 1
    ? '0. 【读取历史守护优先候选】查看系统提示中【你之前的身份推理表】：哪些玩家的 reason 含"守护优先级：高"或"守护优先级：中"？将其列为今晚守护候选起点（若该目标在禁止连守限制内或已出局，改选次高优先候选）'
    : '0. 【首夜】无历史守护记录——直接根据场上信息判断守护目标';
```

**变更 3：思维链升级（静态列表 → Step 0 + 3 步编号）**

改前：
```
【守护优先级】已跳身份的预言家 > 重要神职 > 被狼针对的好人
```

改后：
```
【守护思维链】
${guardHistoryStep}
1. 【守护优先级】已跳身份的预言家 > 重要神职 > 被狼针对的好人
2. 【禁连守处理】若上方历史优先候选恰好是连守禁止对象，顺延至次高优先候选
3. 【最终决策】确定今晚守护目标——若切换了历史优先候选目标，在 thought 中说明切换原因
```

**变更 4：identity_table 写指导升级**

改前：
```
- 守护候选/已守护者：...reason 追加写"N[X]夜守护→[平安夜/成功拦截刀]"
- 疑似神职（但未守护过）：reason 写"行为疑似[预言家/守卫]，守护优先级：高/中"
```

改后：
```
- 守护候选/已守护者：...reason 追加写"${guardNightLabel}夜守护→[平安夜/成功拦截刀]"
- 疑似神职（但未守护过）：reason 写"行为疑似[预言家/守卫]，守护优先级：高/中"（下轮 Step 0 将直接从此读取）
```
