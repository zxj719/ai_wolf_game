## 问题分析

### 竞选发言对跳的双重盲点

竞选期间（SHERIFF_SPEECH 阶段）存在两个结构性盲点：

**盲点 1：claimHistory 未录入竞选声明**

```
Wolf（候选人B） → SHERIFF_SPEECH → 声称"我是预言家，夜1查了3号，查杀"
            ↓
  调用 askAI(B, SHERIFF_SPEECH) → res.speech 记入日志
  ❌ res.claims 从未被解析 → claimHistory 无 jump_seer 记录
            ↓
Seer（候选人C） → SHERIFF_SPEECH → claimHistory 空 → 无对跳感知
```

**盲点 2：预言家竞选发言无对跳反制提示**

即使 claimHistory 有数据（Day 2+ 重选等场景），seer ssHint 也未利用，仅给出固定三步框架。

### 修复路径

```
Wolf 竞选 → res.claims = [{type:'jump_seer',...}]
           → recordClaim({day, playerId, type:'jump_seer', ...})
           → gameState.claimHistory 实时更新

Seer 竞选（后发言）→ ssCounterClaimants 非空
                  → seerSsCounterHint 注入："直接报出真实夜1查验结果"
                  → 真实数据 >> 构造数据（Wang 2025 信息权威原理）
```

## 实现细节

### 1. aiPrompts.js — SHERIFF_SPEECH block

新增 2 个变量（在 `guardSsHint` 之后、`const ssHint` 之前）：

```javascript
// R134: 竞选发言对跳检测
const ssCounterClaimants = playerRole === '预言家'
    ? (gameState.claimHistory || [])
        .filter(c => c.type === 'jump_seer' && c.playerId !== currentPlayer?.id)
        .map(c => c.playerId)
    : [];
const seerSsCounterHint = ssCounterClaimants.length > 0
    ? `\n⚡【对跳情境】${ssCounterClaimants.join('/')}号已在竞选中声称是预言家——核心反制：直接报出真实夜1查验结果（查了X号，金水/查杀），真实数据是唯一无法构造的信息差，自信阐明即可，无需花篇幅争论谁是真的。`
    : '';
```

预言家 ssHint 末尾追加 `${seerSsCounterHint}`，狼人 ssHint ① 追加 claims 声明指引，输出 JSON 新增 claims 字段。

### 2. useDayFlow.js — SHERIFF_SPEECH 后 recordClaim

```javascript
// 柱三：竞选发言中的身份声明即时写入 claimHistory（R134）
if (Array.isArray(res?.claims) && recordClaim) {

        if (!cl || typeof cl !== 'object' || !cl.type) return;
        const { type, ...payload } = cl;
```

## 重要教训（→ LEARNINGS.md）

1. **R134-A**：SHERIFF_SPEECH 竞选声明 claimHistory 盲点根因 — `useDayFlow.js` 竞选循环未调用 `recordClaim`，导致对跳声明不进入 claimHistory，后续候选人和 DAY_SPEECH 均无法结构化感知。修复：在 SHERIFF_SPEECH 循环后解析 `res.claims` 调用 `recordClaim`。
2. **R134-B**：`recordSpeech` 幂等性冲突规避 — `RECORD_SPEECH` 以 `(day, playerId)` 为唯一键，竞选候选人在 day 1 说话 → 同日 DAY_SPEECH 会被 first-write-wins 覆盖。本轮只录入 `claims`（`RECORD_CLAIM` 幂等键是 `(day, playerId, type)`，更细粒度，无此冲突）。若将来需要录入竞选发言内容到 speechHistory，需先修改 RECORD_SPEECH dedup 逻辑（例如加 `phase` 字段）。
3. **R134-C**：T18 测试窗口 600→700 — useDayFlow 调用块约 622 chars，600 窗口刚好截断 `recordClaim,`。**铁律：测试 useDayFlow 参数传入时，窗口至少设 700**（调用块比 useSpeechFlow 更长）。
4. **R134-D**：SS_WINDOW 升级铁律 — 每次 SHERIFF_SPEECH 新增后运行：`node -e "const s=require('fs').readFileSync('src/services/aiPrompts.js','utf8'); const a=s.indexOf('case PROMPT_ACTIONS.SHERIFF_SPEECH:'); const b=s.indexOf('case PROMPT_ACTIONS.SHERIFF_VOTE:',a); console.log('block:',b-a,'余量:',7500-(b-a))"`；余量 < 500 时升级至 8500。
