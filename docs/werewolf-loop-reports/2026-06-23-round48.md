## 本轮优化内容

### 1. getDreamweaverNightPrompt 死函数删除（~129 行）

**背景**：R14 报告中记载"已删除 getDreamweaverNightPrompt"，但实际函数体从未被删除。

**为何死而不知**：
- R14 删除了 `NIGHT_DREAMWEAVER` 的委托路由，改为 `aiPrompts.js` 内联实现
- 但 `dreamweaver.js` 中 `getDreamweaverNightPrompt` 函数体 + `DREAMWEAVER_PROMPTS.nightAction` export 均保留
- 函数通过 export 看起来是"活的"，但实际上生产中无任何代码调用 `getRoleModule('摄梦人').nightAction()`
- 唯一的引用路径是 `DREAMWEAVER_PROMPTS.nightAction`，但这个对象字段本身就是死引用

**删除内容**：
- `getDreamweaverNightPrompt` 函数（原 83-211 行，约 129 行）
- `DREAMWEAVER_PROMPTS.nightAction: getDreamweaverNightPrompt`

### 2. 死解构清理

#### knight.js — `getKnightDaySpeechPrompt`

```js
// 修改前
const { hasUsedDuel, seerChecks, deathHistory, speechHistory } = params;

// 修改后
const { hasUsedDuel } = params;
```

- `seerChecks`：roleParams 中确实存在，但骑士发言框架完全不引用（骑士只看公开发言，不直接用预言家查验数据）
- `deathHistory`、`speechHistory`：从未在 roleParams 中传递，永远是 undefined

#### dreamweaver.js — `getDreamweaverDaySpeechPrompt`

```js
// 修改前
const { dreamHistory, lastDreamTarget, nightDeaths, seerChecks } = params;

// 修改后
const { dreamHistory, lastDreamTarget } = params;
```

- `nightDeaths`：从未在 roleParams 中传递（白天死亡信息已在 `getBaseContext(ctx)` 的 `lastNightInfo` 中覆盖）
- `seerChecks`：roleParams 中存在，但函数体从未引用

### 3. Item 100 委托模式参数合同静态检查

新建 `src/services/__tests__/delegateParams.test.js`，10 个测试：

| 测试 | 目标 |
|------|------|
| T1 | NIGHT_MAGICIAN 所有传给 nightAction 的 key 均被 getMagicianNightActionPrompt 解构 |
| T2 | getMagicianNightActionPrompt 解构含 hasRevealed（R46 修复验证） |
| T3 | NIGHT_MAGICIAN case 传了 hasRevealed |
| T4 | knight.js 已清理死解构 deathHistory/speechHistory |
| T5 | knight.js 保留 hasUsedDuel |
| T6 | dreamweaver.js 已清理死解构 nightDeaths/seerChecks |
| T7 | dreamweaver.js 保留 dreamHistory/lastDreamTarget |
| T8 | getDreamweaverNightPrompt 已从 dreamweaver.js 删除 |
| T9 | DREAMWEAVER_PROMPTS 不含 nightAction 键 |
| T10 | aiPrompts.js NIGHT_DREAMWEAVER 是内联实现（无 nightAction 委托调用） |

**技术难点**：`extractDestructuredKeys` 最初用正则 `/\{([^}]+)\}\s*=\s*params/`，但会匹配函数体 `{`（箭头函数 `=> {` 的 `{`），导致 T1/T5/T7 三个测试提取错误的 key。

**修复**：改为锚点定位法：
1. 先找 `'} = params'` 定位解构块终点
2. 再用 `lastIndexOf('const {', endIdx)` 反向找解构块起点
3. 提取两者之间的内容，去除行注释后 split 分析
