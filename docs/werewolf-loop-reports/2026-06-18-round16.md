## 2. 本轮优化详情

### 优化 1：NIGHT_WOLF multiWolfHint 角色分化一致性

**问题背景**

Round 15 在 `ROLE_DAY_SPEECH_PROMPTS['狼人']` 中加入了详细的角色分化策略（主动方/低调方，基于 Wang 2025 role differentiation 理论）。但 NIGHT_WOLF 的 `multiWolfHint` 只保留了旧版的"立场分散"原则，未引用角色分化框架。

这意味着：狼人在夜间做刀口决策时，读到的协作提示是"白天立场分散"——却看不到具体的角色分工规划。AI 无法在夜间提前意识到"我应该是主动方还是低调方"，导致白天发言时角色分工可能偏离夜间计划。

**修改内容**

位置：`src/services/aiPrompts.js`，`NIGHT_WOLF case` 的 `multiWolfHint` 模板字符串末尾

**改动前**（末尾）：
```
白天各狼须分散发言立场，不同时力挺同一好人（立场高度一致是队友关系暴露的最常见信号）。
```

**改动后**（末尾加一行）：
```
白天各狼须分散发言立场，不同时力挺同一好人（立场高度一致是队友关系暴露的最常见信号）。
白天角色分化（与白天发言策略一致）：先开口的担任【主动方】（质疑好人/带话题方向），后开口的担任【低调方】（装中立评委，"我也在观察"语气，保留信任度）。
```

**博弈论依据**：Wang 2025 的 role differentiation 策略要求两狼在夜间就确定分工角色，而非临场决定——夜间规划 → 白天执行是正确的时序。

### 优化 2：狼人 identity_table 战略化（NIGHT_WOLF + DAY_SPEECH）

**问题背景**

`identity_table` 的完整生命周期：
1. AI 在每次响应中输出 `identity_table`（`{玩家号: {suspect, confidence, reason}}`）
2. `identityTablesRef.current[player.id]` 存储最新版本
3. 每次新请求时以 `previousIdentityTable` 注入系统提示，供 AI 参考并更新
4. **从不在任何 UI 中渲染**（`grep -rn "identity_table" src/components/` 无结果）

这意味着 identity_table 是 AI 的**纯私有跨轮记忆**，可以存储任何形式的战略注记。

**问题**：当前对狼人的 identity_table 只有泛化指令"记录身份推理"，与好人的提示无差异。但狼人已知所有人身份，他们的 identity_table 应有不同的战略用途：
- 不需要"猜测角色"
- 需要追踪威胁等级（谁是高优先刀口）
- 需要维持队友的"公开叙事"一致性（避免在白天发言中和之前的 identity_table 矛盾）

**修改内容**

**位置 1**：`NIGHT_WOLF case` 的思维链之后、输出 JSON 之前，新增：
```
【identity_table 战略用途（你已知所有人真实身份）】
- 对高威胁好人（神职/发言锋利/已对队友起疑）：confidence 填 85-95，reason 写"高优先刀口：[具体威胁原因]"，供下轮夜间决策时优先锁定
- 对已知狼队友：suspect 填"村民"，confidence 填 40-60，reason 填中性描述（维持与你公开发言立场的一致性）
- 对低威胁村民：confidence 填 30-50，reason 填"无明显神职特征，刀口优先级低"
```

**位置 2**：`ROLE_DAY_SPEECH_PROMPTS['狼人']` 的 Step5 之后、输出 JSON 之前，新增：
```
【identity_table 填写策略（日间公开视角，但 reason 可记录私有战略注记）】
- 对高威胁好人（神职/正在推你队友者）：reason 写"刀口候选：[威胁描述]"，便于夜间决策时延续优先级
- 对已知队友：suspect 填"村民"，confidence 填 40-60，reason 填普通描述（维持公开叙事一致）
- suspect 字段统一用好人视角书写，不能直接填"狼人"（即使你已知）
```

**注意**：第三条"不能直接填'狼人'"防止了 sanitizeIdentityTable 可能的副作用，同时也防止 AI 在公开记录中意外标注出队友真实身份。
