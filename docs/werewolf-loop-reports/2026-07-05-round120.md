## 改动细节

### 旧行为（R119 之前，约 8/8 角色，守卫走通用 fallback）

```
守卫 → `2. 【投票策略】有查杀 → 跟投查杀！...好人分票=助狼人优势。`
```

问题：
- 守卫有零间接私有信息（直接守护记录 guardHistory），但 DAY_VOTE 中完全忽略
- 未建立"守护对象=高可信好人"的排除锚点
- 未利用"多次守护频次"信息推断优先保护目标

### 新行为（R120）

**新增 7 个变量**（插入在 `magicianVoteStrategy` 之后、`// R71` 之前）：

```javascript
// 守卫 DAY_VOTE 读写闭环（R120）
const dvGuardHistory = playerRole === '守卫'
    ? (gameState.guardHistory || []).filter(g => g.targetId !== null && g.targetId !== undefined)
    : [];
const dvGuardProtectedAliveIds = [...new Set(
    dvGuardHistory.map(g => g.targetId).filter(id => alivePlayers.find(p => p.id === id))
)];
const dvGuardFreqMap = {};
dvGuardHistory.forEach(g => { dvGuardFreqMap[g.targetId] = (dvGuardFreqMap[g.targetId] || 0) + 1; });
const dvGuardLastTarget = dvGuardHistory.length > 0
    ? dvGuardHistory[dvGuardHistory.length - 1]?.targetId
    : null;
const dvGuardProtectedSummary = dvGuardProtectedAliveIds.length > 0
    ? dvGuardProtectedAliveIds.map(id => `${id}号(${dvGuardFreqMap[id] || 1}次)`).join('、') + '（已确认盟友，从投票选项中排除）'
    : '暂无存活守护记录';
const dvGuardLastTargetRef = dvGuardLastTarget !== null && dvGuardLastTarget !== undefined
    ? `${dvGuardLastTarget}号（昨夜守护对象）`
    : '（昨夜空守）';
const guardVoteStrategy = `2. 【守卫投票—守护排除框架（R120 读写闭环）】
   a) 🛡️守护锚点：存活守护记录：${dvGuardProtectedSummary}；昨夜：${dvGuardLastTargetRef}
   b) 排除逻辑：你多次守护的玩家是你主动判断的关键好人——投票时优先将其从目标列表中排除（confidence 最高，维持队友保护链完整）
   c) 投票优先排序：① 预言家查杀（外部验证，最高信任）→ ② 从未被你守护、identity_table 嫌疑最高的玩家（高嫌疑 + 无守护信任 = 双重标记）→ ③ 发言逻辑崩塌者
   d) 残局推断：若你守护了X号而X号安然无恙，可推测X号是昨夜狼人刀口目标——X号=已验盟友，投票时作为排除锚点使用`;
```

**return 链新增守卫分支**（在魔术师之后、通用 fallback 之前）：
```javascript
  : playerRole === '守卫'
  ? guardVoteStrategy
  : `2. 【投票策略】...`
```

**设计依据**：
- `guardHistory` 是零间接信息（直接守护记录）：与 `seerChecks`（一手验证）同级
- 守护频次 `dvGuardFreqMap` 量化保护优先级（多次守护 = 核心盟友）
- 排除锚点优于通用 fallback：守卫不需要从 identity_table 猜测，有直接记录
- Wang 2025 arxiv:2408.17177：信息丰富角色应以私有信息锚定联盟行为，而非通用猜测

## 本轮教训（→ LEARNINGS.md）

1. **守卫 DAY_VOTE 读写闭环完成（R120-A）**：`dvGuardHistory` 使用 `playerRole === '守卫'` 门控，`dvGuardProtectedAliveIds` 用 `Set` 去重（同一玩家可被保护多夜），`dvGuardFreqMap` 记录频次（多次守护=核心盟友）。排除锚点策略是守卫信息利用最直接的方式（零间接推断）。**设计原则：守卫和预言家一样拥有零间接私有信息，应用相同的"直接读取结构化数据 >> NLP 推断"原则**。
2. **DAY_VOTE 窗口级联规律（R120-B）**：每次 DAY_VOTE block 新增超过约 800 chars，`输出JSON格式:` 就可能超出原窗口。**铁律：每次新增代码后，用 `node -e "const s=require('fs')...; const jsonIdx=block.indexOf('输出JSON格式:'); console.log('offset:', jsonIdx, '< window?', jsonIdx < WINDOW)"` 检查所有依赖 getDayVoteBlock() 的测试文件的窗口值**。当前需检查：R71(16000)、R72(16000)、R119(16000)、R120(16500)。
3. **R64 obsolete test 更新模式（R120-C）**：当早期轮次的"禁止性"测试（`expect(X).toBe(false)`）被后续轮次颠覆时，必须将其更新为对应的"正向确认"测试（`expect(X).toBe(true)`）。不更新会导致历史测试在正确实现后持续失败，混淆回归测试报告。
4. **守卫 DAY_VOTE vs NIGHT 信息利用差异（R120-D）**：NIGHT_GUARD 中读取 identity_table 的"守护优先级：高/中"（R38/R39，即写入的夜间计划）；DAY_VOTE 中直接读取 `gameState.guardHistory` 的结构化记录（直接执行结果）。**设计原则：NIGHT 侧利用 identity_table（AI 自行规划的记忆），DAY_VOTE 侧利用 gameState 结构化数据（系统记录的真实行为）——二者互补，各自独立**。
