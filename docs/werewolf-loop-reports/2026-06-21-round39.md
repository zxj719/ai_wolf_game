## 2. 问题诊断

### 根因：NIGHT_WITCH identity_table 的读写闭环缺失

**发现方式**：执行 R38 LEARNINGS 规则中的诊断命令：
```bash
grep -n "供下轮\|下轮用\|下轮参考\|下轮复查" src/services/aiPrompts.js
```

输出：
```
1448: - ...（下轮 Step 0 将直接从此读取）    ← NIGHT_WOLF，已有读取步骤 ✅
1540: - ...，本夜[是否用毒]；下轮复查"       ← NIGHT_WITCH，无对应读取步骤 ❌
```

**信息流断裂（同 R38 模式）**：
```
N1 夜写：reason = "毒药优先候选：发言带节奏，本夜未用毒；下轮复查"  ← 写入 identity_table
N1 夜后：identity_table 存入 identityTablesRef                        ← 正常存储
N2 夜始：系统提示注入 previousIdentityTable                           ← 正常传递
N2 夜思维链：【用药策略】直接进入解药/毒药/风险/临界四条原则           ← 从不提及"检查历史候选"
                                                                      ← 感知-执行分裂
```

**修复后**：
```
N2 夜 Step 0："查看系统提示中【你之前的身份推理表】：哪些玩家 reason 含'毒药优先候选'？
              将其列为今晚开毒候选起点"       ← 显式读取指令，形成闭环
```

**依据**：
- R38 LEARNINGS：identity_table 的 write 端指导 ≠ read 端执行框架，"供下轮...优先" 是愿景陈述，不是可执行指令
- R26 教训：参数注入解决感知，策略框架才解决执行
- Wang 2025 §4.3：多代理欺骗需要全局策略（跨夜毒药候选积累）+ 局部感知（当晚执行）同时具备

**变更 2：新增两个 const（R18 规范 + 读写闭环）**
```javascript
// R18 规范：避免在 return 模板字符串内直接使用 ctx.dayCount
const witchNightLabel = `N${ctx.dayCount}`;
// 读写闭环（同 R38 NIGHT_WOLF 模式）：首夜无历史；N2+夜从 identity_table 读取"毒药优先候选"
const witchHistoryStep = ctx.dayCount > 1
    ? '0. 【读取历史毒药候选】查看系统提示中【你之前的身份推理表】：哪些玩家的 reason 含"毒药优先候选"？将其列为今晚开毒候选起点（若该目标已死或局面改变，重新评估其他高威胁项）'
    : '0. 【首夜】无历史毒药候选记录——直接根据当前情况判断用药';
```

**变更 3：思维链升级（原 4 要点 → Step 0 + 4 步编号）**

改前：
```
【用药策略】
- 解药考量：被刀者是否为关键神职？...
- 毒药考量：只有高度确信某人是狼且逻辑完全崩坏时才考虑开毒
- 风险评估：毒错好人会导致阵营崩盘
- 临界决策：在危急时刻，保守会导致失败，必须果断出手
```

改后：
```
【用药策略（思维链）】
${witchHistoryStep}
1. 解药考量：被刀者是否为关键神职？...
2. 毒药考量：结合上方历史候选（Step 0），只有高度确信某人是狼且逻辑完全崩坏时才考虑开毒
3. 风险评估：毒错好人会导致阵营崩盘
4. 临界决策：在危急时刻，保守会导致失败，必须果断出手
```

**变更 4：identity_table 写指导升级**

改前：
```
- 本晚被刀目标：...reason 写"N[X]夜被狼刀：..."
- 高威胁毒药候选：...reason 写"毒药优先候选：...；下轮复查"
- 确认出局玩家（已毒杀）：...reason 写"N[X]夜毒药处决，出局确认"
```

改后：
```
- 本晚被刀目标：...reason 写"${witchNightLabel}夜被狼刀：..."
- 高威胁毒药候选：...reason 写"毒药优先候选：...(下轮 Step 0 将直接从此读取)"
- 确认出局玩家（已毒杀）：...reason 写"${witchNightLabel}夜毒药处决，出局确认"
```
