## 三、本轮改动

### 功能：对手面板智能排序

**背景与动机**

SelectScreen 的对手历史面板（展开后 `opp-history-panel`）当前按 `CHARS` 数组的固定顺序（诚→Elza→菲比→Ross→铁蛋→丫→莹）渲染，与玩家的实际胜负状态无关。玩家每次打开面板都需要扫描全部 7 个 chip 才能找到「今天该打谁」——推荐对手可能排在列表末尾。

加入智能排序后，「推荐」对手（胜率≥60%，有历史数据支撑的可赢对象）自动置顶，玩家打开面板即见最优选择，无需扫视。「劲敌」（胜率<35%）仍可见但不显眼，「NEW」（从未对战）置底作为探索提示。

**设计决策**

| 设计点 | 选择 | 理由 |
|--------|------|------|
| 排序依据 | `oppWinRateMap`（全局胜率，不按选中角色过滤） | 与已有「推荐/劲敌」标签数据源一致，避免 UI 矛盾 |
| 四档优先级 | 推荐(0) > 中立(1) > 劲敌(2) > NEW(3) | 语义清晰，劲敌仍出现在 NEW 之前（有数据比无数据好） |
| 同级顺序 | 稳定排序，保持 CHARS 原始顺序 | JS sort 在现代引擎已是稳定的，等级相同时无歧义 |
| 逻辑位置 | `localBoard.js` 纯函数 `sortOppChars` | 可单独测试，与组件解耦，延续 localBoard 工具函数模式 |
| 组件使用 | `useMemo([seenOpps, oppWinRateMap])` | 仅当对战记录变化时重新排序（通常每场结束后才变） |

**四档定义**

| 优先级 | 条件 | 含义 |
|--------|------|------|
| 0 · 推荐 | 已见过 + 胜率 ≥60% | 多次打赢，对当前能力是合适挑战 |
| 1 · 中立 | 已见过 + 35%≤胜率<60% | 旗鼓相当，无明确推荐/回避 |
| 2 · 劲敌 | 已见过 + 胜率 <35% | 历史上较难打，需谨慎或强化 |
| 3 · NEW | 未见过 / total=0 | 从未对战，全新体验 |

## 四、实现路径

### `src/modules/tennis/localBoard.js`

新增 `sortOppChars(chars, seenOpps, oppWinRateMap)` 纯函数：

```js
export function sortOppChars(chars, seenOpps, oppWinRateMap) {
  const getPriority = (c) => {
    if (!seenOpps.has(c.n)) return 3;
    const data = oppWinRateMap[c.n];
    if (!data || data.total === 0) return 3;
    const rate = data.wins / data.total;
    if (rate >= 0.6) return 0;
    if (rate < 0.35) return 2;
    return 1;
  };
  return [...chars].sort((a, b) => getPriority(a) - getPriority(b));
}
```

### `src/modules/tennis/__tests__/localBoard.test.js`

新增 7 个测试用例（`describe('sortOppChars')`）：
- 空数组 → 空数组
- 全部未见 → 保持原始顺序
- 推荐置顶（70% 推荐，50% 中立，20% 劲敌 → B,C,A）
- NEW 排在劲敌之后
- 四档完整顺序：推荐→中立→劲敌→NEW
- 同优先级内保持原始顺序（稳定排序）
- seen 但 total=0 → 视同 NEW（priority=3）
- 不改变原始 chars 数组（纯函数验证）

### `src/modules/tennis/components/SelectScreen.jsx`

1. 导入 `sortOppChars`
2. 新增 `sortedOppChars` useMemo（依赖 `seenOpps`, `oppWinRateMap`）
