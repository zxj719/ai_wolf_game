## 三、本轮改动

### 改动：结算屏动态点评

**问题背景**

R93 的「下轮计划」第 1 项：「结算屏动态点评」——根据胜负/对手/角色/ACE/Clutch 从预写库随机取一句评语，零后端，零 migration，对「赛后仪式感」贡献巨大。

**设计原则**

| 设计点 | 选择 | 理由 |
|--------|------|------|
| 纯前端字符串模板 | `fill(tpl, vars)` 函数替换 `{key}` | 零后端调用，<1ms，无网络延迟 |
| 优先级分层 | ACE≥3 > Clutch≥3 > Counter≥3 > 分数模式 > 通用池 | 最戏剧性的因素优先展示 |
| 角色专属台词 | 7 个角色各 2 条胜/负台词 | 反映角色性格（Elza 狡黠 / 铁蛋憨厚 / 丫懒散…） |
| 合并池随机 | 通用 + 分数池 + 角色池合并后随机选 | 避免某类评语永远不出现 |
| 稳定性 | `useMemo(() => …, [])` 挂载时计算一次 | 结算屏内不重新随机，翻页不变 |

**评语库结构（56 条）**

| 类别 | 条数 | 触发条件 |
|------|------|---------|
| ACE_WIN_LINES | 3 | 胜 + aces≥3（优先级最高） |
| CLUTCH_WIN_LINES | 3 | 胜 + clutchWins≥3 |
| COUNTER_WIN_LINES | 3 | 胜 + countersWon≥3 |
| SWEEP_LINES | 3 | 胜 + 2-0（合并入通用池） |
| COMEBACK_LINES | 3 | 胜 + 2-1（合并入通用池） |
| GENERIC_WIN_LINES | 3 | 胜，基础池 |
| CHAR_WIN_QUIPS | 14 | 胜 + 角色匹配（合并入通用池） |
| ACE_LOSS_LINES | 2 | 败 + aces≥2（优先级最高） |
| CLUTCH_LOSS_LINES | 2 | 败 + clutchWins≥2 |
| CLOSE_LOSS_LINES | 3 | 败 + 1-2（合并入败局池） |
| CRUSHING_LOSS_LINES | 3 | 败 + 0-2（合并入败局池） |
| CHAR_LOSS_QUIPS | 14 | 败 + 角色匹配（合并入败局池） |
| **合计** | **56** | |

**实现路径**

#### `src/modules/tennis/commentary.js`（新建）

- 56 条评语字符串常量（分 12 个命名数组）
- `fill(tpl, vars)` — 替换 `{key}` 占位符
- `pickFrom(arr, vars)` — 随机选一条并 fill
- `getPostMatchCommentary(params)` — 优先级分层选句主函数

#### `src/modules/tennis/components/ResultScreen.jsx`

- `import { getPostMatchCommentary } from '../commentary'`
- 新增 `postComment` useMemo（从 matchStats 取 aces/clutchWins/countersWon）
- `.ending-hero` 卡片 `<p className="comment">` 后追加 `<p className="post-match-comment">{postComment}</p>`

#### `src/modules/tennis/tennis.css`

- 新增 `.ending-hero .post-match-comment`：琥珀色（`var(--accent)`）斜体，14.4px，居中

#### `src/modules/tennis/__tests__/commentary.test.js`（新建）

13 个测试用例覆盖：
- 基础返回类型验证（非空字符串）
- ACE/Clutch/Counter 优先级分支（×5 次迭代确保随机池正确）
- 2-0 sweep 池验证（×30 次采样确认含横扫 lines）
- 2-1 comeback 池验证（×30 次采样确认含逆袭 lines）
- 败局 ACE/Clutch/1-2/0-2 分支覆盖
- 7 个角色全覆盖（无抛出且返回非空）
- 模板变量替换正确（无残留 `{key}` 占位符）
- ACE 优先级高于 Clutch（aces=5 + clutchWins=5 → 返回 ACE 评语）
