## 三、本轮改动

### 功能：被克制累计计数徽章（`bt-counter-lost-count`）

**背景与动机**

至 R139，克制反馈链已有：
- 绿色全屏脉冲（效果拔群 ✅ R137）
- 红色全屏脉冲（被克制 ✅ R139）
- 「💡 招式 A 被招式 B 克制」单球提示（✅ 既有）
- 教练提示「对手惯用招式 ×N，出 X 可克制」（✅ R138）

唯一缺失的是**跨球累计感知**：玩家知道「刚才被克了」，却不知道「我今天已经被克 5 次了」。人类对频率的感知往往低估——需要一个显式的累计数字才能触发换招动机。

**实现（纯 UI，零 reducer 改动）**

`matchStats.counterLost` 由 `battleReducer` 在每次 `RESOLVE` 时准确追踪（已有测试覆盖），`RallyLog` 已持有 `state`，直接读取。

```jsx
// BattleScreen.jsx — RallyLog 函数内，在 bt-counter-tip 之后
{r.counterMul < 1 && !r.win && state.matchStats.counterLost >= 2 && (
  <p className="bt-counter-lost-count">
    本局被克制 ×{state.matchStats.counterLost}
    {state.matchStats.counterLost >= 4 ? ' — 读招是打破僵局的关键！' : ' — 换招试试？'}
  </p>
)}
```

**触发条件**

| 场景 | 显示内容 |
|------|---------|
| 第 1 次被克制 | 不显示（避免首次惊吓） |
| 第 2–3 次被克制 | `本局被克制 ×N — 换招试试？` |
| 第 4+ 次被克制 | `本局被克制 ×N — 读招是打破僵局的关键！` |

首次不显示的设计源于渐进披露原则：初次被克制可能是随机情况，不值得打扰；只有反复出现才说明是系统性问题。

**CSS（tennis.css）**

```css
.tennis-scope .bt-rallylog .bt-counter-lost-count {
  font-size: .71rem; color: rgba(239,68,68,.7); margin-top: 2px;
  animation: bt-tip-in .35s ease both;
}
```

复用既有 `bt-tip-in` 动画（淡入+轻微上滑），与 `bt-counter-tip` 风格一致；颜色与 `fx-redflash` 同调（`rgba(239,68,68,.7)`），不额外引入视觉噪声。
