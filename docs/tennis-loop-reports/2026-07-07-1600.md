## 三、本轮改动

### 功能：LocalBoard 克制效率子榜（「🧩 读招王」）

**背景与动机**

R141 引入了 `CounterDuelBar`——在结算屏展示单局克制对决数据。但该数据从未持久化到本地历史记录，用户无法在家族榜维度看到「谁的读招能力最强」。

本轮将克制数据纳入 `saveLocalRecord`，并在家族榜增加子榜，形成「单局战报 → 历史积累 → 家族排行」完整闭环。

**数据流打通（三层）**

```
battleReducer.js
  matchStats.countersWon  (克制得分，已有)
  rallyCount              (总拍数，已有)
      ↓ (via TennisRoute.jsx onSingleMatchOver → lastMatchStats)
ResultScreen.jsx
  saveLocalRecord({ ..., countersWon, totalRallies })  ← 本轮新增传参
      ↓
localBoard.js
  rec.cw = countersWon ?? 0
  rec.tr = totalRallies ?? 0   ← 本轮新增字段
      ↓
computeCounterEfficiency(records)   ← 本轮新增函数
      ↓
Leaderboard.jsx LocalBoard「读招王」子榜  ← 本轮新增展示
```

**`localBoard.js` 变更**

1. `saveLocalRecord` 新增可选参数 `countersWon`、`totalRallies`，分别存为 `cw`、`tr` 字段（旧记录无此字段，自动被后续过滤跳过，无需迁移）
2. 新函数 `computeCounterEfficiency(records, { minRallies = 10 } = {})`:
   - 只统计 `tr >= minRallies` 的记录（过滤少拍噪音）
   - 按玩家聚合：累计 `cw` 和 `tr`
   - 返回 `[{ name, face, eff, cw, tr }]` 按效率降序，同效率时克制次数多者优先
   - 无满足条件的记录时返回空数组

**`ResultScreen.jsx` 变更**

```jsx
// 修改前
saveLocalRecord({ player: p, opp: o, setsP: state.setsP, setsO: state.setsO });

// 修改后
saveLocalRecord({
  player: p, opp: o, setsP: state.setsP, setsO: state.setsO,
  countersWon: matchStats?.countersWon ?? 0,
  totalRallies: matchStats?.rallyCount ?? 0,
});
```

`matchStats` 已有 prop（R141 起传入），`?.` 链式调用兼容旧入口（无 matchStats 时安全降级为 0）。

**`Leaderboard.jsx` 变更**

在 `LocalBoard` 中：
- 导入 `computeCounterEfficiency`
- 计算 `counterBoard = computeCounterEfficiency(records)`（从全量记录算，不受角色筛选影响）
- 当 `counterBoard.length >= 2` 时，在主表下方渲染「🧩 读招王」子榜（Top-5，4列：排名/选手/克制率/克制÷总拍）
- 副标题注释：`仅统计 10 拍以上对局 · 克制得分 ÷ 总拍数`

**展示条件：`>= 2` 人有效数据**

家庭游戏场景多为少数家庭成员轮流上阵，要求至少 2 人有数据才显示，避免只有 1 人时「自己和自己比」的无意义榜单。
